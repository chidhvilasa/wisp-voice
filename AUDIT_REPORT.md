# Audit Report

Full codebase audit + bug hunt across `src/lib`, `src/hooks`, `src/store`, `src/pages`, `src/overlay`, and `server/src`. Every file in those directories was read in full against nine audit areas (WebRTC engine, VAD, audio pipeline, room management, useVoice hook, signaling server, stores, UI pages, overlay). Only real, reachable bugs were fixed — theoretical/impossible scenarios are documented at the bottom as "investigated, not a bug."

## Bugs found and fixed

| # | File | Bug | Fix | Tested |
|---|------|-----|-----|--------|
| 1 | `src/lib/webrtc.ts` | `connect()` had no re-entrancy guard. Calling it twice (e.g. a double-click on "Join" or React re-render race) while already connecting/connected would open a second signaling WebSocket and a second set of peer connections, silently corrupting engine state. | Added a guard at the top of `connect()` that throws if `connectionState` is `'connecting'` or `'connected'`. | Yes |
| 2 | `src/lib/webrtc.ts` | `handleSignalMessage`'s `'joined'`/`'offer'`/`'answer'` cases called `initiateOffer()`/`handleOffer()`/`handleAnswer()` with a bare `void`, discarding the returned promise. Any rejection (e.g. `createOffer()` throwing) became an unhandled promise rejection instead of surfacing to the app. | Replaced `void fn(...)` with `fn(...).catch((error) => this.emitNegotiationError(error))`, which emits a normal `'error'` event consumers already listen for. | Yes |
| 3 | `src/lib/webrtc.ts` (`attachRemoteStream`), `src/lib/audio.ts` (`init`), `src/lib/vad.ts` (`init`) | All three places create a `new AudioContext()` and start using it immediately. Browsers/webviews commonly create `AudioContext` in a `'suspended'` state under autoplay policy, meaning audio nodes are silently dead until something calls `.resume()` — local mic, remote peer audio, and VAD/speaking-indicator detection could all silently fail to produce any sound or signal. | After each `new AudioContext()`, added `if (ctx.state === 'suspended') { void ctx.resume?.() }` (optional-chained so it's a no-op against the test suite's lightweight mocks). | Yes |
| 4 | `src/lib/webrtc.ts` (`cleanupPeer`) | When a peer left, `cleanupPeer()` removed the peer's `gainNode` but never removed its entry from `peerVolumes`. In a long-running session with many people cycling through a room, `peerVolumes` would grow without bound. | Added `this.peerVolumes.delete(peerId)` to `cleanupPeer()`. | Yes |
| 5 | `src/lib/webrtc.ts` (`sendChat`, `sendRoomLocked`) | `RTCDataChannel.send()` can throw synchronously (e.g. `TypeError` if the payload exceeds `maxMessageSize`, or if the channel closes mid-call). Neither call site caught this, so one bad/oversized chat message to one peer would throw out of `sendChat()`, breaking delivery to every other connected peer in the same loop iteration. | Wrapped each `channel.send(payload)` call in try/catch with a `console.warn`, so a failure to one peer no longer aborts the loop for the rest. | Yes |
| 6 | `src/lib/rooms.ts` (`createRoom`) | `fetch()` had no timeout. If the signaling server's `/room` endpoint hangs (dead Worker, network black hole), `createRoom()` would hang forever with no way for the UI to recover or show an error. | Added an `AbortController` with a 10s timeout wrapped around the `fetch()` call; an abort is converted to the existing friendly "Unable to create a room" error. | Yes |
| 7 | `src/lib/rooms.ts` (`saveRecentRoom`, `removeRecentRoom`) | Both called `localStorage.setItem()` with no error handling, unlike `getRecentRooms()` which already had a try/catch. A full or disabled `localStorage` (private browsing, quota exceeded) would throw out of these "best-effort" persistence calls and break the join/create flow that calls them. | Wrapped both `setItem` calls in try/catch, silently swallowing failures since this data is non-critical, best-effort persistence. | Yes |
| 8 | `src/store/settingsStore.ts` | The `persist` middleware used zustand's default `merge`, which is a **shallow** merge (`{ ...currentState, ...persistedState }`). If a user's saved settings from an older app version had an incomplete `hotkeys` object (e.g. missing a newly-added hotkey added in a later release), rehydration would overwrite the entire `hotkeys` object with the incomplete persisted one, leaving newer hotkeys completely undefined instead of falling back to their defaults. | Added a custom `merge` option that deep-merges `hotkeys` specifically (`{ ...currentState.hotkeys, ...persisted.hotkeys }`), so missing keys fall back to current defaults. | Yes |
| 9 | `src/overlay/Overlay.tsx` (`handleMouseDown`) | The drag handler attached `mousemove`/`mouseup` listeners to `window` but only removed them inside `handleUp` (fired on mouse-up). If the Overlay component unmounted while a drag was in progress (window closed, app state change), the listeners were never removed — a permanent leak of two `window`-level listeners, with the dangling `handleMove` continuing to call `win.setPosition()` on a potentially-destroyed window indefinitely. | Added a `dragCleanupRef` that stores a detach closure whenever listeners are attached, and a `useEffect` unmount cleanup that calls it, removing any still-active drag listeners. | Yes |
| 10 | `server/src/index.ts` | No CORS headers anywhere, and no `OPTIONS` preflight handling. The Tauri webview's origin differs from the deployed Cloudflare Worker's `*.workers.dev` domain, so without `Access-Control-Allow-Origin`, the browser/webview blocks JS from reading the `POST /room` response — meaning **"Create Room" would always fail in real production use**, not just as an edge case. | Added a `CORS_HEADERS` object applied to every response (including 400/404 error responses), plus an `OPTIONS` preflight handler returning `204` with those headers. | Yes |

## Improvements beyond strict bug fixes

- The CORS fix's regression tests also assert CORS headers are present on **error** responses (404), not just the happy path, since a partial fix (success-only headers) would have been an easy way to "pass" a shallow test while still breaking error handling for legitimate cross-origin error inspection.
- The AudioContext-resume test deliberately asserts on **both** AudioContexts created by `attachRemoteStream` (the engine's own gain-staging context and the per-peer `VADProcessor`'s context), since both were silently affected by the same root cause and a narrower test would have missed the VAD-side instance.
- Documented the full set of investigated-but-not-bugs below, so the absence of a fix for those is a documented decision rather than an oversight.

## Audit areas investigated and judged NOT bugs (no fix made)

These were each checked against the actual code/tests, not assumed:

- **webrtc.ts**: `setPeerVolume` for a nonexistent peerId is already safe (a `Map` set is fine regardless of whether the peer exists yet); `sendChat` with zero open channels still fires the local echo correctly; `disconnect()` is null-safe when `localStream` was never set; `handleSignalMessage`'s `JSON.parse` is already try/caught; stats-loop interval is already cleared in `disconnect()`.
- **vad.ts**: `destroy()` is idempotent and `analyze()` already guards on a missing `analyser`/`buffer`; double `init()` is safe because `init()` calls `destroy()` first.
- **audio.ts**: `setOutputVolume(0)` followed by `setDucked(false)` correctly restores to the new base of `0` (tracked as a separate field), not `1`; `destroy()` is fully idempotent.
- **rooms.ts**: `joinRoom()`'s re-entrancy risk shares the same root cause as bug #1 (`engine.connect()` itself now throws), so no separate fix was needed.
- **useVoice.ts**: no additional bugs found beyond the shared `connect()` re-entrancy root cause already fixed in `webrtc.ts`; unmount-during-connect, reconnect-backoff-post-unmount, and Tauri listener unregistration were all already handled correctly.
- **server/src/index.ts**: rate limiter correctly resets per new peerId on reconnect; peerId collision is a 16^8 keyspace (negligible in practice); malformed/very long room codes pose no injection risk (no DB — Cloudflare's `idFromName` accepts arbitrary strings safely); single-client-thousands-of-connections DoS is an infrastructure-level concern (Cloudflare rate limiting / WAF), not an application-code fix.
- **voiceStore.ts**: `reset()` already explicitly clears `chatMessages`; `emitSafe()` already wraps Tauri `emit()` in try/catch; `peers` Map growth is structurally bounded by the engine's max-peer enforcement plus `reset()` on disconnect.
- **Home.tsx / Settings.tsx / devices.ts**: empty device lists render gracefully; hotkey-capture listeners are correctly cleaned up via effect dependencies; `getDevicesByKind()` already wraps `enumerateDevices()` in try/catch; soundboard's ~2MB practical limit is already disclosed in `KNOWN_ISSUES.md` as a known limitation, not a silent-failure bug.
- **Room.tsx**: chat auto-scroll always force-scrolls on new messages regardless of current scroll position — real but minor/cosmetic UX nit, not something that "will genuinely break," so left as-is per the audit's explicit scope.

## Verification

- `npx vitest run` — **47/47 tests pass** (35 pre-existing + 12 new regression tests: 9 in `src/lib/stress.test.ts`, 3 in `server/src/index.test.ts` for the CORS fix).
- `npx tsc --noEmit` (root) — clean, zero errors.
- `npx tsc --noEmit` (`server/`) — clean, zero errors.
- `npm run build` — succeeds. Output: `dist/assets/index-tCI9T3DL.js` 278.16 kB (gzip 85.40 kB), `dist/assets/index-DtX328Kw.css` 29.66 kB (gzip 5.97 kB).

## Final verdict

**Production ready for friends-scale use: yes.** All ten identified bugs are real, fixed, and covered by regression tests; the most production-critical one (missing CORS headers, which would have completely broken "Create Room" against any deployed Worker) is now resolved, and the remaining nine harden long-running-session edge cases (reconnection, audio autoplay policy, leak cleanup, error surfacing) without changing any user-facing behavior. The app's own `KNOWN_ISSUES.md` is honest about out-of-scope limitations (stub noise suppression, free TURN relay, no Rust build verified in this environment) that don't block friends-scale use but should be tracked before a wider release.
